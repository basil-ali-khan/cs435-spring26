# Assignment 5 — TEE-Style Confidential Loan Scoring Simulation

> **Note:** This is a **software simulation** of a Trusted Execution Environment (TEE) workflow. It does not use real hardware enclaves (Intel SGX, AMD SEV, AWS Nitro, etc.). The purpose is to demonstrate the attestation-then-decrypt pattern in Python.

---

## What This Does

A bank wants to run a loan scoring model in a confidential environment. Before sending sensitive applicant data, the client needs proof that the *correct, unmodified* scoring code is running.

This simulation implements that workflow:

1. **Measurement** — compute a SHA-256 hash of `scoring_module.py` (the "enclave measurement").
2. **Attestation** — compare the live hash to a pre-registered expected hash stored in `config.json`.
3. **Conditional decrypt** — only if the hash matches, decrypt the Fernet-encrypted applicant payload.
4. **Safe scoring** — run rule-based scoring inside the protected function; return only the score and decision.
5. **Tamper detection** — modifying `scoring_module.py` causes a different hash, which triggers a block before any decryption occurs.

## How to Run

### Step 1 — Register the expected hash

Run this **once** (or whenever you intentionally update `scoring_module.py`):

```bash
python generate_config.py
```

Expected output:
```
[CONFIG] Hashed: scoring_module.py
[CONFIG] SHA-256: <64-char hex digest>
[CONFIG] Saved expected hash to config.json
[READY]  You can now run run_normal.py and run_tampered.py
```

### Step 2 — Normal run (attestation passes)

```bash
python run_normal.py
```

Processes 4 synthetic applicants. Each payload is encrypted client-side, sent to `protected_service()`, verified, decrypted inside the protected boundary, and scored. Raw applicant fields are never printed.

### Step 3 — Tampered run (attestation blocked)

```bash
python run_tampered.py
```

The script temporarily overwrites `scoring_module.py` with malicious code that unconditionally approves every application. The protected service detects the hash mismatch, refuses to decrypt, and returns a `BLOCKED` result. The original file is always restored at the end (even if the script errors).

## How the Tamper Case Was Tested

1. `generate_config.py` was run to record the SHA-256 of the clean `scoring_module.py`.
2. `run_tampered.py` overwrites `scoring_module.py` with a version that always returns `APPROVED`.
3. `protected_service()` recomputes the hash of the on-disk file — it no longer matches `config.json`.
4. The service returns `BLOCKED` and never calls `Fernet.decrypt()`.
5. The original file is restored via `finally` block, so subsequent runs are unaffected.

## What the Simulation Captures vs. Does Not Capture

| Aspect | This Simulation | Real TEE (e.g. Azure CVM) |
|---|---|---|
| Code integrity check | SHA-256 of a file on disk | Hardware-measured hash of enclave memory at load time |
| Attestation authority | Self-checked (same process) | Remote attestation service (MSFT Attestation, AMD KDS, etc.) |
| Key provisioning | Fernet key passed as argument | Key released by KMS only after attestation report is verified |
| Memory isolation | None (same OS process) | Hardware enforces isolation from hypervisor / OS kernel |
| Side-channel protection | None | Partial (hardware mitigations vary by platform) |
| Replay protection | None | Nonces and timestamps in attestation reports |

The simulation correctly models the **pattern**: measure → verify → decrypt. It does not provide actual hardware isolation or cryptographic attestation proofs.

## Hyperscaler Connection — Azure Confidential Computing

### How It Works in Production

**Azure Confidential VMs** (and **Azure Confidential Computing** with SGX-backed enclaves) provide hardware-enforced TEEs where the guest VM or enclave memory is encrypted and isolated even from the underlying hypervisor, Azure infrastructure operators, and the host OS. Code running inside an AMD SEV-SNP Confidential VM can generate a hardware attestation report — a cryptographically signed statement produced by the CPU itself — that includes a measurement of the boot firmware, OS, and workload. A client or key-management service verifies this report against AMD's Key Distribution Service (KDS) before releasing encryption keys. This means sensitive data (like applicant loan records) can only be decrypted inside a VM whose exact code state has been verified by hardware, which is exactly the pattern this simulation demonstrates in software.

**Azure Attestation** is the managed service that handles the verification side: it validates SGX quotes or SEV-SNP attestation reports, checks them against a policy (e.g., expected code hash), and returns a signed JWT that an application can use to prove its integrity to external parties. Combined with **Azure Key Vault** with Confidential VM support, the decryption key for applicant payloads would only be released once the attestation JWT is presented — providing a hardware root of trust that no software-only simulation can fully replicate.
